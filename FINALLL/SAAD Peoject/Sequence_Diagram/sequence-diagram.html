Full description

This sequence diagram describes how the student, system, registrar, and finance department interact over time during the course registration process. The flow begins when the student logs in and the system displays the available courses. The student then selects a course and submits a registration request, which the system forwards to the registrar to check seat availability. If the course is full, the system informs the student and the process repeats so another course can be selected. If seats are available, the system contacts the finance department to verify the student’s payment status. When payment is incomplete, the system asks the student to retry and the loop continues. When payment is confirmed, the registration is completed successfully, and the process ends. This diagram emphasizes how registration is only finalized when both seat availability and financial clearance are confirmed.

Key elements

  • Actors & lifelines: User, Client/UI, API Gateway, Service Layer, Database.
  • Messages: Synchronous calls (solid arrows), returns (dashed), and async events (dotted).
  • Activation bars: Show when a participant is processing.

How to use

  1. Read top → down (time axis); participants are laid out left → right.
  2. Label messages with actual endpoints and expected latencies for performance checks.
  3. Use the diagram to locate bottlenecks, race conditions, and where to add tracing/logging.

Practical notes

When preparing this diagram for developers or reviewers, annotate key messages with endpoint names (e.g., POST /login), token exchange steps, and retry/error handling flows.
