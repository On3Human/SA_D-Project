Use Case Diagram for System Analysis and Design Project

What is a Use Case Diagram?

A Use Case Diagram is a behavioral diagram from the Unified Modeling Language (UML) that illustrates how users interact with a system. It captures the functional requirements of a system by showing the different ways users (actors) can interact with the system to achieve specific goals (use cases). Essentially, it provides a high-level view of the system's functionality and scope.

It is a crucial tool in the early stages of system analysis and design, helping stakeholders understand the system's boundaries, its primary functions, and the roles of its users.

Key Components of a Use Case Diagram

  • Actor: Represents a role that a user plays when interacting with the system. An actor can be a human user, another system, or an external device. It is typically drawn as a stick figure.
  • Use Case: Represents a specific functionality or a sequence of actions performed by the system that yields an observable result of value to a particular actor. It is typically drawn as an oval.
  • System Boundary: A rectangle that encloses the use cases and defines the scope of the system. Anything inside the boundary is part of the system, while actors are outside.
  • Associations (Relationships): Lines connecting actors to use cases, indicating that an actor participates in a use case.
  • Relationships Between Use Cases:
    • Include (<<include>>): Indicates that one use case incorporates the functionality of another use case. The included use case is necessary for the base use case to complete.
    • Extend (<<extend>>): Indicates that one use case adds functionality to another use case under certain conditions. The extending use case is optional and augments the base use case.
    • Generalization: Represents an inheritance relationship where a specialized use case inherits the behavior and meaning of a more general use case.

Purpose and Benefits

  • Requirements Elicitation: Helps in identifying and documenting functional requirements from the user's perspective.
  • Communication: Provides a clear and simple way to communicate system functionality to stakeholders, including non-technical ones.
  • Scope Definition: Clearly defines the boundaries and scope of the system, helping to prevent scope creep.
  • Test Case Generation: Use cases can be directly translated into test cases, ensuring that all defined functionalities are tested.
  • System Design: Serves as a foundation for more detailed design activities, such as class diagrams and sequence diagrams.

Example: Online Shopping System

Let's consider a simplified online shopping system to illustrate a use case diagram.

Actors:

  • Customer: A person who buys products.
  • Administrator: A person who manages products and orders.

Use Cases:

  • View Products: Customer can browse available products.
  • Add Product to Cart: Customer can add items to their shopping cart.
  • Remove Product from Cart: Customer can remove items from their shopping cart.
  • Checkout: Customer completes the purchase process.
    • <<include>> Process Payment: Part of the checkout process.
    • <<include>> Generate Order: Part of the checkout process.
  • Manage Products: Administrator can add, update, or delete products.
  • Manage Orders: Administrator can view and update order statuses.
  • <<extend>> Apply Discount: Optionally extends "Checkout" under specific conditions.

Relationships:

  • Customer associates with View Products, Add Product to Cart, Remove Product from Cart, Checkout.
  • Administrator associates with Manage Products, Manage Orders.
  • Checkout includes Process Payment.
  • Checkout includes Generate Order.
  • Checkout extends Apply Discount.
